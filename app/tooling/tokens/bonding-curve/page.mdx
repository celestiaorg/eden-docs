## Overview

A bonding curve defines how the price of a token changes as supply increases or decreases. In a **linear** bonding curve, the price rises at a constant rate with supply, making the math predictable and gas-efficient to compute directly onchain.


## Why use bonding curves?

Bonding curves make token pricing automatic, transparent, and trustless.

Instead of guessing at a “fair” launch price, the contract itself sets the price mathematically from the first block. Each trade adjusts the price along a curve, there are no order books, no market makers, no waiting for an exchange.

Projects like Pump.fun proved how effective this model can be: tokens launch with onchain liquidity, predictable price movement, and verifiable math that anyone can inspect.

Bonding curves turn price discovery into code, replacing speculation with a clear, deterministic rule.

## How linear bonding curves work

A bonding curve defines **price as a function of total supply**. When someone mints tokens, total supply increases and price rises along the curve. When tokens are burned, supply decreases and price falls.

![Linear bonding curve](/images/guides/bonding-curve/linearBondingCurve.png)


A **linear** curve means price changes at a constant rate with supply. The function is:

$$
p(s) = \text{initialPrice} + (\text{slope} \times s)
$$

where:

* $s$ is the current total supply
* $\text{initialPrice}$ is the base price when supply is zero (optional, can be set to 0)
* $\text{slope}$ determines how much price increases per token minted

## The math

When tokens are minted or burned, calculating the cost is a little tricky due to the fact that each incremental mint pushes the price up (and each incremental burn pushes the price down). One possible solution is to loop through each tiny fraction of a token, adding up prices as you go. But, ERC-20 tokens have 18 decimals, meaning even a small mint would require billions++ of steps, and the EVM would run out of gas long before it finished.

Instead of looping, we treat the entire mint or burn as one smooth move along a straight line, and the total cost to mint is just the **area under that line**.

![Linear bonding curve](/images/guides/bonding-curve/totalCost.png)


At first, that shape looks like a trapezoid, and you probably remember from geometry class the formula area of a trapezoid is $A = \frac{1}{2} \times (b_1 + b_2) \times h$, but we can make it easier than that. If you slice off the small triangle at one end and flip it over, it fills the gap on the other side and turns the trapezoid into a perfect rectangle. So the total cost is simply the **average price** times the **number of tokens**.

![Linear bonding curve](/images/guides/bonding-curve/totalCostArrow.png)

<br/>
$$
\begin{align*}
\text{totalCost} &= \text{height} \times \text{width} \\\\
                 &= \text{averagePrice} \times \text{numberOfTokens} \\\\
                 &= \frac{p_\text{start} + p_\text{end}}{2} \times \text{numberOfTokens}
\end{align*}
$$


That’s it. No loops, no calculus, just the midpoint of a line times its width. Burning works the same way but in reverse: the same rectangle, just sliding back down the line.


## Implementing the contract

Now that the geometry is clear, let’s wire it into Solidity. Below are the key functions from a minimal ERC‑20 linear bonding‑curve token. A complete project (contracts, tests, and a small UI) is available on [GitHub](TODO).

### Price helpers

These two helper functions calculate the token’s price along the curve at any given supply. `_priceAt()` is used internally, while `getCurrentPrice()` exposes the current onchain price for external calls or front‑ends.

```solidity
function _priceAt(uint256 supply) internal view returns (uint256) {
    return initialPrice + (slope * supply) / 1e18;
}

function getCurrentPrice() public view returns (uint256) {
    uint256 a = slope;      // wei per token^2
    uint256 s = totalSupply();
    uint256 b = initialPrice;
    return (a * s) / 1e18 + b;
}
```

<Callout type="info">
ERC‑20 amounts use **18 decimals** (atomic units). Both `slope` and `supply` are 18‑decimal‑scaled. Multiplying them temporarily produces **36 decimals** of precision. Dividing by `1e18` brings the result back to the standard **18‑decimal** scale so the price is expressed in atomic units (wei). 

Both `_priceAt()` and `getCurrentPrice()` return **atomic units**. To display a human‑readable price in a UI, divide the return value by `1e18`.

</Callout>

### Mint cost (midpoint × numberOfTokens)

This function implements the midpoint‑pricing formula. It calculates the total cost to mint a given number of tokens based on the average of the start and end prices. It uses the `Math.mulDiv` function to avoid overflow and ensure the result is in atomic units.

```solidity
function calculateCost(uint256 amount) public view returns (uint256) {
    require(amount > 0, "amount=0");
    uint256 s = totalSupply();
    uint256 pStart = _priceAt(s);
    uint256 pEnd   = _priceAt(s + amount);
    // ((pStart + pEnd) * amount) / (2 * 1e18), rounded up
    return Math.mulDiv(pStart + pEnd, amount, 2 * 1e18, Math.Rounding.Ceil);
}
```

<Callout type="info">

`Math.mulDiv(x, y, denominator, rounding)` computes `(x * y) / denominator` with full 512‑bit precision to avoid overflow, then applies rounding.

* **x** = `pStart + pEnd` (sum of endpoint prices, 18d)
* **y** = `amount` (numberOfTokens, 18d)
* **denominator** = `2 * 1e18` → `2` averages the endpoints; `1e18` corrects decimal scale (36d → 18d)
* **rounding** = `Ceil` → buyers never underpay by 1 wei

Because all three terms are scaled, the extra `/ 1e18` inside the denominator is essential to keep the result in 18‑decimal atomic units. 

</Callout>

### Minting tokens

This public function lets users mint tokens by sending TIA to the contract. It verifies payment, enforces slippage limits, and refunds any excess sent beyond the required cost.

```solidity
function mintTokens(uint256 amount, uint256 minTokenOut) public payable nonReentrant {
    require(amount > 0, "Cannot mint zero tokens");

    uint256 cost = calculateCost(amount);
    require(msg.value >= cost, "Insufficient ETH sent");
    require(amount >= minTokenOut, "Slippage: fewer tokens than expected");

    _mint(msg.sender, amount);

    if (msg.value > cost) {
        uint256 refund = msg.value - cost;
        (bool ok, ) = payable(msg.sender).call{value: refund}("");
        require(ok, "Refund failed");
    }
}
```

* `nonReentrant` protects the transfer + state sequence.
* `minTokenOut` provides simple slippage protection against adverse price movement between quote and execution.
* Any excess TIA is returned after mint.

### Quoting tokens for a fixed TIA amount

This helper inverts the bonding‑curve equation to determine how many tokens a buyer can receive for a given TIA input. It solves the pricing curve algebraically using the quadratic formula.

```solidity
function calculateTokensForTia(uint256 ethAmount) public view returns (uint256) {
    require(ethAmount > 0, "Must send more than 0 TIA");

    uint256 s = totalSupply();
    uint256 a = slope;
    uint256 b = initialPrice;
    uint256 c = ethAmount;

    uint256 A = (a / 2);               // quadratic term
    uint256 B = ((a * s) / 1e18) + b;  // linear term
    uint256 C = c;                     // constant term

    require(A > 0, "Slope too small");

    uint256 discriminant = (B * B) + (4 * A * C);
    uint256 sqrtDiscriminant = Math.sqrt(discriminant);

    uint256 numerator = sqrtDiscriminant - B;
    uint256 denominator = 2 * A;
    require(denominator > 0, "Denominator is zero");

    // Scale by 1e18 to return an 18‑decimal token amount
    uint256 amount = (numerator * 1e18) / denominator;
    require(amount > 0, "Not enough ETH to buy tokens");
    return amount;
}
```

This makes it easy for a frontend to quote *tokens‑for‑TIA* in one call without looping through each incremental mint.

### Refund for burns

When users burn tokens, the contract calculates how much ETH they receive back from the bonding‑curve reserve. It uses the same midpoint‑pricing math as `calculateCost`, but in reverse. It mirrors `calculateCost` but uses `Floor` so the contract cannot over‑refund.


```solidity
function calculateRefund(uint256 amount) public view returns (uint256) {
    uint256 s = totalSupply();
    if (amount == 0) return 0; // allow 0 without revert
    require(amount <= s, "Cannot burn more tokens than the current supply");

    uint256 pLow  = _priceAt(s - amount);
    uint256 pHigh = _priceAt(s);
    // ((pLow + pHigh) * amount) / (2 * 1e18), rounded down
    return Math.mulDiv(pLow + pHigh, amount, 2 * 1e18, Math.Rounding.Floor);
}
```

### Burning tokens

The `burnTokens` function lets holders redeem tokens for TIA directly from the bonding‑curve reserve. It ensures the contract has enough balance, then performs the burn and sends the refund.

```solidity
function burnTokens(uint256 amount) public nonReentrant {
    if (amount == 0) { return; }            // no‑op allowed
    require(balanceOf(msg.sender) >= amount, "Insufficient balance to burn");

    uint256 refund = calculateRefund(amount); // compute before burning
    require(address(this).balance >= refund, "Contract has insufficient ETH for refund");

    _burn(msg.sender, amount);

    (bool ok, ) = payable(msg.sender).call{value: refund}("");
    require(ok, "Refund transfer failed");
}
```

* Calculates the refund **before** burning to lock in price at the current supply.
* Verifies the reserve covers the payout.
* Uses `nonReentrant` for safety.


## Summary

The contract and its math together implement a fully deterministic linear bonding curve. Minting adds area under the line, burning removes it, and both use the same equations.

Because the curve is linear, all calculations are closed-form and gas-efficient — no loops, no approximations, just onchain geometry.
