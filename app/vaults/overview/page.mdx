# Vaults overview

import { Callout } from 'nextra/components'

Vaults let users pool assets into a smart contract and receive shares. The vault runs a strategy, yield flows back, and share price increases. Users deposit assets, get shares, and redeem shares for more assets than they put in.

Think of vaults as programmable, permissionless yield. They don't have to live fully onchain—you could pool funds to buy a condo, list it on Airbnb, and flow rental income back into the vault. Shareholders own a slice of both the asset and the profits.

## ERC-4626: Synchronous vaults

[ERC-4626](https://eips.ethereum.org/EIPS/eip-4626) provides a standard interface for instant deposit/redeem vaults. Every vault exposes the same methods: `deposit`, `mint`, `redeem`, `withdraw`.

ERC-4626 vaults are also ERC-20 tokens. The vault contract extends ERC-20, so when you deposit, the vault mints you shares. Those shares are implemented inside the vault itself—you can call `balanceOf`, `transfer`, `approve`, all the usual ERC-20 methods.

### How vault accounting works

Instead of tracking each user's balance with for-loops (gas-inefficient), vaults track two numbers:

- `totalAssets`: Total assets controlled by the vault
- `totalSupply`: Total shares issued

Each share's value is implied by the ratio:

```
pricePerShare = totalAssets / totalSupply
```

**Example:**

Ten users each deposit 100 USDC.

- `totalAssets = 1000`
- `totalSupply = 1000`
- `pricePerShare = 1000 / 1000 = $1`

After a month, the vault earns 100 USDC profit.

- `totalAssets = 1100`
- `totalSupply = 1000`
- `pricePerShare = 1100 / 1000 = $1.10`

If you redeem 100 shares, you get $110 back instead of the $100 you deposited. The contract never updated your balance directly—it just updated two numbers. No loops, gas-efficient.

### Core methods

**Deposits:**

- `deposit(assets)`: Deposit assets, receive shares
- `mint(shares)`: Mint exact number of shares, spending required assets

**Withdrawals:**

- `redeem(shares)`: Burn shares, receive assets
- `withdraw(assets)`: Withdraw exact assets, burning required shares

**Notes:**

- `.asset()` must be one pre-defined ERC-20 token. ERC-4626 is single-asset only.
- [ERC-7575](https://eips.ethereum.org/EIPS/eip-7575) aims to generalize to multi-asset vaults.
- If a user is timelocked, `maxRedeem`/`maxWithdraw` should return 0

## ERC-7540: Asynchronous vaults

[ERC-7540](https://eips.ethereum.org/EIPS/eip-7540) extends ERC-4626 with request-based flows for strategies that can't settle instantly onchain.

Sometimes deposits require KYC checks. Assets live across a bridge and you're waiting for finality. Redemptions are gated by validator exit queues. Or the vault buys off-chain collateral and waits for settlement.

Yield doesn't always live in a synchronous world.

### Lifecycle: request → pending → claim

Instead of minting shares right away, you first make a request. That request is marked pending while external processes happen. Once ready, status flips to claimable, then you mint shares.

**Deposits:**

1. User calls `requestDeposit(assets)` → status: pending
2. Fulfillment triggered by operator or external system (KYC, settlement, etc.) → status: claimable
3. User calls `deposit(assets)` → shares minted

**Redemptions:**

1. User calls `requestRedeem(shares)` → status: pending
2. Fulfillment triggered by operator or external system → status: claimable
3. User claims assets

**Why no `requestMint` or `requestWithdraw`?**
The time between request and fulfillment is unpredictable. Specifying exact outputs upfront would be misleading.

### Use cases

- **RWA**: Real-world asset tokenization with compliance checks
- **LSTs/LRTs**: Liquid staking/restaking with validator exit queues
- **Cross-chain**: Deposits requiring bridge finality
- **Off-chain collateral**: Strategies buying physical assets (real estate, commodities)
