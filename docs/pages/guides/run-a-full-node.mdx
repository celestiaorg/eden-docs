# Run a full node on Eden mainnet

This guide walks through running an Eden mainnet full node using Docker Compose. The stack consists
of two services:

- **ev-reth** — the EVM execution layer (handles RPC, transaction processing, and state)
- **ev-node** — the consensus/DA layer (fetches blobs from Celestia and drives ev-reth via the
  Engine API)

## Prerequisites

- Docker and Docker Compose v2 installed
- A running [Celestia light node](https://docs.celestia.org/nodes/light-node) connected to
  Celestia **mainnet**, with its RPC accessible from this machine

**Minimum hardware requirements:**

| Component | Minimum    | Recommended   |
| --------- | ---------- | ------------- |
| CPU       | 8 cores    | 16 cores      |
| RAM       | 16 GB      | 32 GB         |
| Storage   | 500 GB SSD | 1 TB NVMe SSD |
| Network   | 1 Gbps     | 10 Gbps       |

## Download the artifacts

The Eden mainnet deployment package lives in the
[`eden-docs` repository](https://github.com/celestiaorg/eden-docs), so artifact version updates are
atomic with the documentation.

Clone the repository and enter the artifacts directory:

```bash
git clone --depth 1 https://github.com/celestiaorg/eden-docs
cd eden-docs/public/eden-artifacts/mainnet
```

The directory contains:

```
compose.yaml
ev-node.genesis.json
ev-reth.genesis.json
entrypoint.fullnode.sh
init-1-ev-node.sh
init-1-ev-reth-snapshot.sh
init-2-ev-node-snapshot.sh
.env
```

## Configure the environment

The `.env` file ships with mainnet defaults. The only required change is your Celestia light node
RPC address:

```bash
# Open .env and set:
DA_ADDRESS=http://your-celestia-light-node:26658
```

The DA namespace variables (`DA_HEADER_NAMESPACE`, `DA_DATA_NAMESPACE`) are pre-set to mainnet
values — do not change them.

Default port assignments for reference:

| Variable                  | Default | Service               |
| ------------------------- | ------- | --------------------- |
| `EV_RETH_RPC_PORT`        | `8545`  | ev-reth HTTP RPC      |
| `EV_RETH_WS_PORT`         | `8546`  | ev-reth WebSocket RPC |
| `EV_RETH_ENGINE_PORT`     | `8551`  | ev-reth Engine API    |
| `EV_RETH_DISCOVERY_PORT`  | `30303` | ev-reth P2P           |
| `EV_RETH_PROMETHEUS_PORT` | `9001`  | ev-reth metrics       |
| `EV_NODE_RPC_PORT`        | `7331`  | ev-node RPC           |
| `EV_NODE_PROMETHEUS_PORT` | `26660` | ev-node metrics       |

:::warning[Prometheus endpoints]
Both ev-reth and ev-node bind their Prometheus metrics to `0.0.0.0` (all interfaces). Restrict
access to these ports at your firewall if this machine is publicly reachable.
:::

## Choose a sync method

:::tip[Recommended]
**Snapshot restore** gets your node operational within minutes rather than syncing from block 0.
:::

### Method A: Restore from snapshot (recommended)

The default `compose.yaml` includes init containers (`init-1-snapshot-ev-reth`,
`init-2-snapshot-ev-node`) that automatically download the latest database snapshots from Binary
Builders storage before starting ev-reth and ev-node. No changes to `compose.yaml` are needed.

Start the stack:

```bash
docker compose up -d
```

Monitor snapshot download progress, then follow ev-node sync:

```bash
docker compose logs -f init-1-snapshot-ev-reth
docker compose logs -f init-2-snapshot-ev-node
docker compose logs -f ev-node
```

### Method B: Sync from genesis

This method starts from block 0. Sync time depends on your connection and hardware.

`jwt-init` is a one-shot init container that generates a shared JWT secret on first run. The secret
authenticates the Engine API connection between ev-node (consensus) and ev-reth (execution) —
ev-reth rejects Engine API calls not signed with the matching secret, so both services must share
the same file before either starts.

Edit `compose.yaml` to remove the snapshot init containers:

1. Delete the `init-1-snapshot-ev-reth` and `init-2-snapshot-ev-node` service blocks.

2. In the `ev-reth` service, update `depends_on` to only require `jwt-init`:

   ```yaml
   ev-reth:
     depends_on:
       jwt-init:
         condition: service_completed_successfully
   ```

3. In the `ev-node` service, update `depends_on` to only require `ev-reth`:

   ```yaml
   ev-node:
     depends_on:
       ev-reth:
         condition: service_started
   ```

Then start the stack:

```bash
docker compose up -d
docker compose logs -f ev-node
```

## Verify the node is running

Check ev-reth is responding to RPC:

```bash
curl -s -X POST -H "Content-Type: application/json" \
  --data '{"jsonrpc":"2.0","method":"eth_blockNumber","params":[],"id":1}' \
  http://localhost:8545
```

Check ev-node health:

```bash
curl -s http://localhost:7331/health/live
```

## Monitor sync progress

```bash
# Follow all service logs
docker compose logs -f

# Follow a specific service
docker compose logs -f ev-node
docker compose logs -f ev-reth
```

Prometheus metrics are available at:

- ev-reth: `http://localhost:9001`
- ev-node: `http://localhost:26660`

| Chain name      | `eden-mainnet`   |
| DA layer        | Celestia mainnet |
| DA start height | `9409947`        |

The DA start height is the Celestia mainnet block from which Eden begins reading blobs. It is a
fixed chain parameter set at genesis and does not change.
